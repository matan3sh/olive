# i18n Conventions

**Purpose:** This project supports English and Hebrew (RTL). There are two distinct i18n layers — static UI strings and dynamic CMS content. This doc defines both and when to use each.

---

## The Two Layers

| Layer | What it covers | Where it lives | How it's used |
|---|---|---|---|
| Static UI strings | Labels, buttons, aria text, empty states, fixed copy | `lang/en.json` + `lang/he.json` | `useTranslations` / `getTranslations` |
| Dynamic CMS content | Product names, descriptions, hero text, nav items | Sanity `localizedString` type | Resolved in `lib/cms/` via `locale` param |

---

## Rules

1. Never hardcode UI text inside a component. Every visible string must come from one of the two layers.
2. Always add new static string keys to **both** `lang/en.json` and `lang/he.json` at the same time. Missing a key in one file will cause a runtime error.
3. Use `useTranslations` in client components (`'use client'`).
4. Use `getTranslations` in async server components (pages, layouts).
5. Namespace keys by feature (e.g., `shop.filters.size`, not `filterSize`). Keep the JSON structure nested to match the UI structure.
6. Dynamic CMS text fields that appear on-site use `type: 'localizedString'` in the Sanity schema and are resolved to the correct locale in `lib/cms/` before being passed to components. Components always receive a plain `string`, never `{ en, he }`.
7. Never pass a `LocalizedString` object to a component. Resolve it in `lib/cms/` using the `locale` parameter.
8. Page metadata (title, description) uses `getTranslations` from the `metadata` namespace in the locale layout.

---

## Patterns

### Adding a new static string

Both files must be updated together:

```json
// lang/en.json
{
  "shop": {
    "filters": {
      "size": "Size"
    }
  }
}
```

```json
// lang/he.json
{
  "shop": {
    "filters": {
      "size": "נפח"
    }
  }
}
```

### Using translations in a client component

```tsx
'use client'
import { useTranslations } from 'next-intl'

export default function FilterBar() {
  const t = useTranslations('shop')

  return <label>{t('filters.size')}</label>
}
```

### Using translations in a server component

```tsx
// app/[locale]/layout.tsx
import { getTranslations } from 'next-intl/server'

export async function generateMetadata({ params }) {
  const { locale } = await params
  const t = await getTranslations({ locale, namespace: 'metadata' })
  return { title: t('title'), description: t('description') }
}
```

### Dynamic CMS content — resolved before hitting the component

```ts
// lib/cms/products.ts
function resolve(p: RawProduct, locale: Locale): Product {
  return {
    ...p,
    title: p.title[locale],       // LocalizedString → string
    description: p.description[locale],
  }
}

export async function getProducts(locale: Locale): Promise<Product[]> {
  const raw = await sanityClient.fetch(PRODUCTS_QUERY)
  return (raw as RawProduct[]).map((p) => resolve(p, locale))
}
```

```tsx
// app/[locale]/shop/page.tsx — locale resolved here, component gets plain strings
const products = await getProducts(loc)  // Product[], not RawProduct[]
return <ShopPage products={products} />
```

```tsx
// components/shop/ShopPage.tsx — receives plain strings, no locale logic needed
interface Props {
  products: Product[]  // title: string, not LocalizedString
}
```

### Locale type

```ts
// lib/cms/types.ts
export type Locale = 'en' | 'he'
export interface LocalizedString { en: string; he: string }
```

Always cast the `locale` param from the URL to `Locale` before passing to CMS functions:

```ts
const { locale } = await params
const loc = locale as Locale
```

## Checklist

- [ ] Are new static strings added to both `lang/en.json` and `lang/he.json`?
- [ ] Are JSON keys nested under the correct feature namespace?
- [ ] Does the client component use `useTranslations`, and the server component use `getTranslations`?
- [ ] Do components receive plain `string` values — never `LocalizedString` objects?
- [ ] Is the locale resolved in `lib/cms/` and cast as `Locale` in the page before being passed down?
- [ ] Is every visible UI string sourced from one of the two layers (no hardcoded text)?
